UIView与UIWindow

UIView主要干的就是两件事：

  1. 呈现内容
  2. 用户交互

这两件工作都涉及一个关键词：层级管理。

视图方面：UIView除了呈现自己的内容外，还要负责管理自己的child view，child view又要管理自己的child view，以此类推，它们层叠起来的这整个视图层级构成了App的完整用户界面。

交互方面：UIView继承自UIResponder。作为事件响应链上的重要环节，它们也遵循响应链的事件传送规则，逐层传递。这自然也与层级管理密不可分。

从数据结构来说，很显然，UIView的视图层级是一颗树，就数据结构课堂上学习到的知识而言，对一棵树的数据访问或者维护都必然是从树根root节点出发，那么UIView的视图层级的树根在哪里呢。
UIWindow，UIWindow继承自UIView，拥有UIView的层理管理能力和事件传递能力，它不呈现自身的任何内容，只是作为UIView的一个容器，被抽象出来负责UIView视图层级的一个统筹协调管理工作。
显然，UIWindow维护着一个UIView视图层级。

  • 绘制视图内容时，UIApplication找到UIWindow，UIWindow再让UIView逐层逐一绘制自己。

  • 事件传递时，UIApplication从系统事件队列中获取一个事件，传递给UIWindow，UIWindow再通过UIView层级逐层传递，直到事件被处理。

The Delivery of Touch Objects Follows a Defined Path
In the main event loop, the application object gets (raw) touch events in its event queue, packages them as UITouch objects in UIEvent objects, and sends them to the window of the view in which the touch occurred. The window object, in turn, sends these objects to this view, which is known as the hit-test view. If this view cannot handle the touch event (usually because it hasn’t implemented the requisite event-handling methods) the event travels up the responder chain until it is either handled or discarded.
